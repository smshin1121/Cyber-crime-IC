# IC Deduplication & Verification Standard

사이버범죄 국제공조 작전의 중복 판정 및 데이터 검증 표준.
**모든 데이터 입력 경로(웹서치, 파일업로드, Excel, Web Clipper, RSS)에서 반드시 적용한다.**

## 1. 적용 시점

| 입력 경로 | 적용 단계 | 비고 |
|---------|---------|------|
| 파이프라인 수집 (ic-pipeline) | evaluator 단계 | Decision Tree 후 자동 적용 |
| Excel 임포트 | 감사(audit) 단계 | 전수 감사 시 적용 |
| Obsidian Web Clipper | 인제스트 전 | raw/ 저장 후 통합 전 적용 |
| 수동 추가 | 페이지 생성 전 | 작성자가 직접 확인 |

## 2. 동일 작전 판정 기준 (5-Point Match)

두 건이 동일한 작전인지 판단할 때 **5가지 기준 중 3개 이상 일치하면 MATCH**로 판정한다.

### 기준표

| # | 기준 | 일치 조건 | 가중치 | 비고 |
|---|------|---------|--------|------|
| 1 | **작전명** | 동일하거나 변형 (Operation Cronos = Op Cronos = Cronos) | 높음 | 작전명이 정확히 일치하면 단독으로 MATCH 가능 |
| 2 | **날짜 근접성** | 시작일/종료일이 6개월 이내 | 중간 | 동일 작전의 다른 phase는 별개 건 |
| 3 | **참여 기관 겹침** | 주요 기관 2개 이상 동일 | 중간 | lead agency 일치 시 가중 |
| 4 | **수치 데이터 일치** | 체포/압수/참여국 수가 ±20% 이내 | 중간 | 소스마다 수치가 다를 수 있음 |
| 5 | **범죄유형 일치** | 동일 crime_type 분류 | 낮음 | 범죄유형만으로는 판별 불가 |

### 판정 매트릭스

| 일치 수 | 판정 | 처리 |
|--------|------|------|
| **5/5 또는 작전명 정확 일치** | **MATCH (high confidence)** | 보충 레퍼런스로 추가 |
| **3-4/5** | **MATCH (medium confidence)** | LLM 에이전트 검증 후 추가 |
| **2/5** | **POSSIBLE** | 수동 확인 필요 → _workspace/review_queue.md에 기록 |
| **0-1/5** | **NO_MATCH** | 신규 작전 또는 제외 |

## 3. 작전명 정규화 규칙

동일 작전이 다른 이름으로 표기되는 패턴:

| 패턴 | 예시 | 처리 |
|------|------|------|
| 접두어 생략 | "Operation Cronos" = "Cronos" = "Op Cronos" | `Operation ` 제거 후 비교 |
| Phase/Wave 구분 | "Cronos Phase 1" ≠ "Cronos Phase 3" | Phase가 다르면 **별개 건** |
| 로마숫자/아라비아 | "HAECHI IV" = "HAECHI 4" | 통일 후 비교 |
| 현지어/영어 | "작전명 크로노스" = "Operation Cronos" | 영어 표준명으로 매핑 |
| 미명명 작전 | "공식 작전명 없음" | 작전명 기준 비교 불가 → 나머지 4기준으로 판정 |
| 작전 시리즈 | Jackal, Jackal III | 시리즈 내 **별개 건** (번호가 다르면 별개) |

## 4. 날짜 비교 규칙

| 상황 | 판정 |
|------|------|
| 동일 연월 | 일치 |
| 6개월 이내 | 일치 (동일 작전의 후속 보도 가능) |
| 6-12개월 | 불확실 → 다른 기준에 가중 판단 |
| 12개월 초과 | 불일치 (같은 이름이라도 별개 wave일 수 있음) |
| 날짜 미상 ("정확한 날짜 미상") | 날짜 기준 비교 불가 → 나머지 4기준 |

## 5. 수치 비교 규칙

| 상황 | 판정 |
|------|------|
| 정확히 동일 | 강한 일치 |
| ±20% 이내 | 일치 (소스마다 집계 차이) |
| ±20% 초과 | 불일치 (단, 후속 보도에서 수치 갱신 가능 → 날짜 확인) |
| 한쪽 미상 | 수치 기준 비교 불가 → 나머지 기준 |
| 0 vs 양수 | 불일치 |

## 6. LLM 매칭 프로토콜

자동 매칭(키워드/퍼지)이 불확실할 때 LLM 에이전트를 사용한다.

### 프롬프트 템플릿

```
다음 두 건이 동일한 국제공조 작전인지 판단하라.

[Excel 데이터]
- 작전명: {op_name}
- 시작일: {start}
- 참여기관: {agencies}
- 참여국가: {countries}
- 범죄유형: {crime}
- 내용: {content}
- 출처 URL: {url}

[위키 페이지]
{wiki_page_content의 frontmatter + Summary 섹션}

5가지 기준으로 판정하라:
1. 작전명 일치 여부
2. 날짜 근접성 (6개월 이내?)
3. 참여 기관 겹침 (주요 기관 2개 이상?)
4. 수치 데이터 일치 (체포/압수 ±20%?)
5. 범죄유형 일치 여부

판정: MATCH / NO_MATCH / POSSIBLE
근거: {각 기준별 판단 이유}
신뢰도: high / medium / low
```

### 비용 최적화

- 먼저 작전명 정확 일치 검사 (O(1))
- 불일치 시 날짜+범죄유형 사전 필터링 (O(n))
- 후보가 남으면 LLM 매칭 (비용 발생)
- 후보가 없으면 바로 NO_MATCH

## 7. MATCH 시 처리

MATCH로 판정되면:

1. **URL을 기존 위키 페이지의 References에 추가**
2. **CI 재산출** (source_count 증가 → CI 상승)
3. **기존 페이지에 없는 새 정보가 있으면 본문 보강** (새 기관명, 추가 수치 등)
4. **log.md에 기록**: `[날짜] update | {작전명} — supplementary reference added from {source}`

### CI 재산출 공식

```
CI = (source_count_score × 0.3) + (max_source_tier_score × 0.4) + (source_diversity_score × 0.3)

source_count_score: 1→1.25, 2-3→2.5, 4-6→3.75, 7+→5.0
max_source_tier_score: Tier1→5, Tier2→4, Tier3→3, Tier4→2
source_diversity_score: 고유 소스 유형 수 (1-5)
```

## 8. 교훈 (Lessons Learned)

### 교훈 1: 퍼지 매칭은 반드시 오매칭을 일으킨다
- **사건**: Tier 4 64건을 퍼지 매칭으로 기존 페이지에 연결 시도 → Operation Avalanche에 Operation Chakra, Hawk 등 무관 URL 37건 혼입
- **원인**: 단어 단위 매칭이 "cyber", "operation" 같은 일반 단어에 반응
- **교훈**: 키워드 매칭은 사전 필터링에만 사용하고, 최종 판정은 5-Point Match 또는 LLM 에이전트가 수행해야 한다
- **적용**: 자동 매칭 → LLM 검증 → 수동 확인 3단계 체계

### 교훈 2: 작전명이 없는 데이터는 매칭이 극히 어렵다
- **사건**: Excel 570건 중 "공식 작전명 없음" 등 비명명 작전이 다수
- **원인**: 소규모 작전, 일반 뉴스 보도에서 작전명을 명시하지 않는 경우
- **교훈**: 작전명 없는 건은 나머지 4기준으로 판정해야 하며, 2기준 이하 일치 시 NO_MATCH가 안전
- **적용**: 미명명 작전은 LLM 매칭 필수

### 교훈 3: 동일 작전의 Phase는 별개 건이다
- **사건**: Operation Cronos Phase 1 / Phase 3, Operation Endgame Phase 1 / Phase 2
- **원인**: 같은 작전명이지만 시점, 참여국, 결과가 모두 다름
- **교훈**: Phase/Wave 번호가 다르면 별개 위키 페이지로 관리
- **적용**: 작전명 비교 시 Phase 번호까지 포함

### 교훈 4: 소스 다양성이 CI보다 중요할 수 있다
- **사건**: Operation Avalanche CI=4.0 (37개 레퍼런스) vs Operation Synergia CI=4.0 (9개 레퍼런스)
- **원인**: 대량의 Tier 4 소스보다 소수의 Tier 1-2 소스가 데이터 품질에 더 기여
- **교훈**: CI가 높다고 데이터가 정확한 것은 아님. max_source_tier가 실질적 품질 지표
- **적용**: CI 외에 source_tier를 독립적 품질 지표로 병행 표시

### 교훈 5: 대량 배치는 단계적 검증이 필수
- **사건**: Excel 700건 일괄 처리 시 Decision Tree → CI → 중복 제거 → LLM 매칭 4단계
- **교훈**: 한 단계에서 모든 것을 판정하려 하면 오류 누적. 각 단계에서 확실한 건만 통과시키고 불확실한 건은 다음 단계로 넘김
- **적용**: INCLUDE → REVIEW → EXCLUDE 3-verdict 체계 유지

### 교훈 6: 되돌리기(revert) 능력이 핵심
- **사건**: 잘못된 퍼지 매칭 적용 → git checkout으로 즉시 되돌림
- **교훈**: 대량 위키 수정 전 반드시 커밋. 오류 발견 시 즉시 revert 가능해야 함
- **적용**: 모든 대량 작업 전 git commit, 작업 후 검증, 문제 시 revert

## 9. 검증 체크리스트

대량 데이터 입력 시 반드시 확인:

- [ ] Decision Tree (Q1-Q4) 통과 여부 확인
- [ ] 모든 출처 URL이 실제 접근 가능한지 확인
- [ ] source_tier 올바르게 분류했는지 확인
- [ ] CI 공식 정확하게 적용했는지 확인
- [ ] 중복 판정 5-Point Match 적용했는지 확인
- [ ] MATCH 건의 URL이 실제 해당 작전을 다루는지 LLM 또는 수동 검증
- [ ] 기존 페이지에 추가한 레퍼런스가 정확한지 확인
- [ ] git commit 후 검증 → 문제 시 revert 준비
- [ ] exclusion_log에 제외 사유 기록
- [ ] 결과 보고서 작성
